Installation

Sahil-code requires Rust 1.75+ and Cargo. Clone the repo and build:

git clone https://github.com/SahilBamb/sahil-code
cd sahil-code
cargo build --release

The binary will be at target/release/claw. Add it to your PATH or symlink it:

sudo ln -s $(pwd)/target/release/claw /usr/local/bin/claw

Prerequisites

  • Rust 1.75+ (install via rustup)
  • An API key for at least one provider (Anthropic, OpenAI, or a running Ollama instance)

Quick Start

Set your API key and start the interactive REPL:

export ANTHROPIC_API_KEY="sk-ant-..."
claw

Or send a single prompt:

claw prompt "add error handling to the parse function in src/parser.rs"

Or use a different provider:

export OPENAI_API_KEY="sk-..."
claw --provider openai --model gpt-4o prompt "explain this codebase"
Sahil-code auto-detects your provider based on which API key environment variable is set. If ANTHROPIC_API_KEY is present, it defaults to Anthropic. If OPENAI_API_KEY is present, it defaults to OpenAI. If OLLAMA_HOST is set, it defaults to Ollama.

Conversation Runtime

The ConversationRuntime is the central engine. It manages the message loop between the user, the LLM, and the tool executor. Each "turn" follows this cycle:

  1. User message is added to the session
  2. Full message history + system prompt is sent to the API provider
  3. The LLM responds with text and/or tool calls
  4. Tool calls are checked against the permission policy
  5. Approved tools are executed, results appended to the session
  6. If there were tool calls, the loop repeats (up to max_iterations)
  7. When the LLM responds with only text (no tool calls), the turn ends

The runtime is generic over both the API client and the tool executor, which is how the same core logic works across all providers and tool implementations.

pub struct ConversationRuntime<C, T> {
    session: Session,
    api_client: C,          // implements ApiClient trait
    tool_executor: T,       // implements ToolExecutor trait
    permission_policy: PermissionPolicy,
    system_prompt: Vec<String>,
    max_iterations: usize,  // default: 16
    usage_tracker: UsageTracker,
}

Providers

Sahil-code abstracts LLM providers behind the ApiClient trait. Any provider that implements this trait can be plugged into the conversation runtime.

Anthropic

Uses the Messages API with streaming via SSE. Supports Claude 3.5 Sonnet, Claude 3 Opus, and all Anthropic models. Set ANTHROPIC_API_KEY to enable.

OpenAI

Uses the Chat Completions API with streaming. Works with GPT-4o, GPT-4, and any OpenAI-compatible endpoint (like Azure, Together, Groq). Set OPENAI_API_KEY to enable.

Ollama

Native integration with Ollama's /api/chat endpoint. Run models locally with zero API costs. Set OLLAMA_HOST (defaults to http://localhost:11434).

Fallback Chains

Configure a fallback provider that activates when the primary fails (rate limits, outages). The FallbackProvider wraps multiple ApiClient instances and tries them in order.

Capability Detection

The detect_capabilities function returns a ModelCapabilities struct for any model, reporting:

  • supports_tools — whether the model can handle tool call schemas
  • supports_streaming — whether SSE streaming is available
  • context_window — max input tokens (e.g., 200K for Claude 3.5)
  • max_output_tokens — max generation length

Tool System

Tools are the agent's hands. Each tool implements the Tool trait:

pub trait Tool: Send + Sync {
    fn spec(&self) -> ToolSpec;
    fn execute(&self, input: &Value) -> Result<ToolOutput, String>;
}

Tools are managed by a ToolRegistry which supports dynamic registration. The default registry includes all 9 built-in tools plus any plugins.

Structured Output

Every tool returns a ToolOutput with a content string and optional metadata (JSON). This allows machine-readable data alongside human-readable summaries.

Middleware Stack

Tool execution flows through a middleware pipeline. Pre-hooks run before execution (can block), post-hooks run after (can transform output). The built-in TimingHook adds duration_ms to every tool result.

pub trait PreHook: Send + Sync {
    fn before(&self, tool_name: &str, input: &Value) -> HookResult;
}

pub trait PostHook: Send + Sync {
    fn after(&self, tool_name: &str, output: &mut ToolOutput);
}

Permissions

The permission system controls what the agent can do. Set via the CLAW_PERMISSION_MODE environment variable:

ModeBehavior
allow-allAll tools execute without prompting (default)
read-onlyOnly read_file, glob_search, grep_search are allowed; everything else is denied
promptEach tool call prompts for interactive approval: allow, deny, or allow for session

Per-tool overrides are supported via PermissionPolicy::with_tool_mode(), allowing fine-grained control like "allow all reads, prompt for writes, deny bash".

Layered Configuration

Configuration is loaded from three layers, with later layers overriding earlier ones:

  1. ~/.claw/config.json — User-global defaults
  2. .claw/config.json — Project-level settings (committed to repo)
  3. .claw/config.local.json — Local overrides (gitignored)

Config Fields

{
  "provider": "anthropic",
  "model": "claude-sonnet-4-20250514",
  "base_url": "https://api.anthropic.com",
  "max_tokens": 4096,
  "permission_mode": "allow-all"
}

CLI flags always take precedence over config files. Environment variables take precedence over config files but are overridden by CLI flags.

Project Instructions

Sahil-code automatically discovers instruction files from the current directory and its ancestors:

  • AGENTS.md
  • CLAUDE.md
  • .claw/instructions.md

These files are appended to the system prompt, giving the model awareness of your project's conventions, architecture, and preferences. Write them in markdown with coding standards, architecture decisions, and things the agent should know.

Git Context Injection

If the current directory is a git repository, sahil-code automatically runs git status --short and git diff --stat and injects the output into the system prompt. This gives the agent awareness of:

  • Current branch
  • Uncommitted changes
  • Staged files
  • Changed file statistics

Sessions

Every conversation is a Session — a vector of ConversationMessage objects. Sessions are auto-saved to ~/.claw/sessions/{id}.json after each turn.

Resuming Sessions

# Resume the most recent session
claw --resume last

# Resume a specific session
claw --resume ~/.claw/sessions/1234567890-42.json

Exporting Sessions

# Export as markdown
claw export ~/.claw/sessions/1234.json --format md

# Export as self-contained HTML
claw export ~/.claw/sessions/1234.json --format html

# Export from the REPL
/export md
/export html

Compaction

As conversations grow, they can exceed the model's context window. Compaction summarizes older messages to free up space while preserving essential context.

Automatic compaction triggers after each turn if should_compact() returns true (i.e., estimated tokens exceed the threshold). Manual compaction is available via the /compact REPL command.

Compaction preserves the most recent messages and replaces older ones with a summary. The CompactionConfig controls the token threshold and how many recent messages to keep.

Cost Tracking

The CostTracker estimates costs based on model-specific pricing (input tokens, output tokens, cached tokens). It tracks per-turn and cumulative session costs.

# In the REPL
/cost

# Output:
# Model: claude-sonnet-4-20250514
# Turns: 5
# Total: $0.0342 (input: $0.0198, output: $0.0144)

Pricing is built-in for major models. Unknown models default to $0 (useful for local Ollama models).

Safety & Sandboxing

Filesystem Sandbox

The FilesystemSandbox restricts all file operations to a configurable root directory (default: CWD). Paths with .. traversal or symlinks that escape the sandbox are rejected. This prevents the agent from reading or modifying files outside your project.

Bash Command Policy

The BashPolicy blocks dangerous command patterns by default:

  • rm -rf /, rm -rf ~
  • sudo
  • curl | sh, wget | sh
  • mkfs, dd if=
  • :(){ :|:& }; (fork bombs)

Custom blocked patterns can be added via BashPolicy::custom().

Audit Trail

The AuditLog records every tool execution to a JSONL file at ~/.claw/audit/. Each entry includes the tool name, input, output, duration, and timestamp.

Rate Limiting

The RateLimiter enforces:

  • Max tool calls per session (default: 200)
  • Max tokens per session (configurable)

Undo / Rollback

The UndoStack saves file contents before write_file or edit_file operations. The /undo command restores the previous state. For newly created files, undo removes them entirely.

Plugins

Plugin Format

Plugins are directories with a plugin.json manifest:

{
  "name": "my-plugin",
  "version": "0.1.0",
  "description": "A custom tool plugin",
  "tools": [
    {
      "name": "my_tool",
      "description": "Does something useful",
      "parameters": { "type": "object", "properties": { ... } }
    }
  ],
  "hooks": ["tool:before", "tool:after"]
}

Plugin Discovery

Plugins are loaded from ~/.claw/plugins/ and .claw/plugins/. The PluginLoader scans these directories for valid plugin.json files and registers their tools into the ToolRegistry.

MCP Client

The built-in MCP (Model Context Protocol) client connects to external tool servers via JSON-RPC over stdin/stdout. Start an MCP server and sahil-code can discover and use its tools automatically.

Hook System

The HookRegistry supports event-driven hooks:

  • tool:before — fires before any tool execution
  • tool:after — fires after any tool execution

Register callbacks that receive the tool name and input/output, enabling custom logging, transformation, or validation logic.

Bundled Skills

Skills are higher-level capabilities built on tools:

  • verify — Run tests and report results
  • commit — Stage and commit with a generated message
  • refactor — Multi-file refactoring with validation
  • explain — Deep code explanation with references

Multi-Agent Orchestration

For complex tasks, the AgentOrchestrator manages multiple sub-agents working in parallel.

Sub-Agent Tasks

Each AgentTask has its own description, tool whitelist, and token budget. The orchestrator tracks task status (Pending, Running, Complete, Failed) and collects results.

File Locks

The FileLockManager prevents concurrent modification conflicts when multiple agents work on overlapping files. Agents acquire locks before editing and release them when done.

Agent Communication

Agents communicate via structured AgentMessage variants:

  • Progress — "editing file 3 of 7"
  • Clarification — escalated to the user
  • FileLock / FileUnlock — coordinate shared resources
  • Complete — task finished with summary

Plan-Execute Pattern

Structured multi-step execution: define a Plan with ordered PlanSteps, review it, then execute step-by-step. Each step tracks its status (Pending, InProgress, Done, Skipped) and output.

Server & API

HTTP API

Start the API server with claw serve --port 8080. Endpoints:

MethodPathDescription
POST/sessionsCreate a new session
POST/sessions/{id}/turnsSend a message (SSE response)
GET/sessions/{id}Get session state
DELETE/sessions/{id}Delete session

SSE Streaming

The /turns endpoint streams events using Server-Sent Events:

  • text_delta — Incremental text from the assistant
  • tool_start — Tool execution began
  • tool_end — Tool execution completed
  • done — Turn complete with usage summary

WebSocket

Bidirectional communication at /sessions/{id}/ws. The client sends WsClientMessage (user messages, tool approvals, cancel), the server sends WsServerMessage (text deltas, tool events, approval requests).

Headless / CI Mode

claw --headless --max-turns 5 "fix all linter errors in src/"

Runs without interactive input. Auto-approves all tools. Outputs a JSON HeadlessResult to stdout with:

  • success — boolean
  • assistant_text — final assistant output
  • tool_calls — array of tool call summaries
  • total_tokens — tokens consumed
  • turns — number of turns executed

Exit code 0 on success, non-zero on failure.

VS Code Extension

A VS Code extension scaffold is included at extensions/vscode/. It provides:

  • Chat panel — Webview-based conversation UI in the activity bar
  • Tool call tree — Real-time view of tool execution status
  • Permission UI — Approve/deny tool calls from the editor
  • Configuration — Server URL, auto-approve read-only, diff preview toggle

Vim Mode

Toggle vim keybindings in the REPL with /vim. Supports:

  • Normal mode — h/l movement, w/b word navigation, 0/$ line start/end, x delete, d delete line, C change to end
  • Insert mode — i, a, A, I entry points
  • Visual mode — v to select, y to yank, d to delete selection
  • Repeat counts — 3l moves right 3, 2w jumps 2 words

Crate Structure

Sahil-code is organized as a Cargo workspace with four crates plus an extension directory:

sahil-code/
  Cargo.toml              # Workspace root
  crates/
    core/                 # Conversation runtime, sessions, permissions,
                          # config, compaction, cost, security, plugins,
                          # agents, server types
    providers/            # Anthropic, OpenAI, Ollama API clients,
                          # SSE parser, fallback, capabilities
    tools/                # 9 built-in tools, Tool trait, ToolRegistry,
                          # middleware stack
    cli/                  # Binary entry point, REPL, arg parsing,
                          # markdown renderer, line editor, vim mode
  extensions/
    vscode/               # VS Code extension (TypeScript)
